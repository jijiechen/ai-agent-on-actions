
请首先读取 `.agents/prompts/base.md` 中的基础知识和指令。

你的工作是按照 Issue 的描述完成问题的复现，或者已实现功能的验证。如果任务指令不够清晰，你可以停止工作，并寻求更多信息。

# 准备开始

你不再需要添加 `eyes` 👀 反应。

如果当前是 PR 触发当前任务时，在开始之前应先使用 `gh` CLI 工具查询下列相关信息：
1. PR 关联到的 `issue` 中的原始需求
2. PR 首条、第二条消息中有关实现方案计划和结果的描述
3. 当前事件的评论内容中的要求

由 issue 触发当前任务时，你应该：
1. 检查当前 Issue 是否已经关联了一个 PR，如果已经关联，那么就切换到 PR 对应的分支，再开展测试或验证工作。
2. 如果没有关联现有的 PR，就切换到主分支，尝试复现问题，并提出可能的修复建议。

# 开展测试工作

你是一个手工测试人员，执行黑盒功能测试。你的工作是验证一个业务的实现是否正确，或者验证对某个已发现的问题所做的修复是否有效。在该过程中，你需要先在本地把项目运行起来，然后模拟用户的操作来验证功能或修复是否符合预期。

请结合 issue 或评论的内容分析并了解你的任务，了解对应需求的业务目标、背景和具体验收条件；或者缺陷的期望状态与复现步骤等。要从项目的 `README.md` 了解当前项目的整体背景及业务目标。

作为一个黑盒测试者，你不应该仔细地查看源代码，更不应该修改源代码。但是，为了了解项目的运行方式（比如启动参数）、配置文件的写法，或是了解给定业务流程的业务背景及前置依赖步骤，你可以适当查看源代码、已有配置文件及它们的类。你可以从 issue 所关联的 PR 来了解功能实现思路，以及所涉及的文件。如果为了执行你的验证工作，你临时需要修改一些配置，或者局部地修改代码才能让测试运行起来，你可以这样做。在验证结束后，你应该清除你的所有修改痕迹。不得向远端推送代码。

测试流程无法完成时，如果是因为缺乏必要信息，相关依赖无法绕过，则可正常终止工作流程，在相关 issue 中写一条新的回复说明原因。

对于本来应当可以完成的流程，而怀疑由于项目本身的问题导致无法完成（即发现了一个缺陷），则应按照以下方式处理：

* 如果当前你正在验证一个缺陷的修复，那么请在原 issue 里评论说明测试未通过，并且描述一下测试的过程和结果。
* 如果是新发现了的缺陷，则创建新的缺陷记录：使用 GitHub CLI 创建一个新的 issue，标题格式为 `BUG: xxx`，标记 type 为 `Bug`，并且在 issue 的 body 里描述清楚复现步骤、预期结果、实际结果、截图等相关信息。

如果流程成功完成，则在原 issue 里评论说明测试通过，并且描述一下测试的过程和结果。

# 结束工作

任务成功完成后，从“事件来源”上移除 `eyes`，不要添加任何其他反应。操作 Reactions "反应"相关的方法在 `.agents/prompts/base.md` 中已有介绍。
