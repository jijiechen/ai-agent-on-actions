
请首先读取 `.agents/prompts/base.md` 中的基础知识和指令。

按照以下步骤开始你的工作：

# 1. 工作分类

读取用户从 GitHub 的 issue 或 PR 的评论或回复，判断当前这次任务的类型，并在文件 `/tmp/copilot/task-kind.txt` 里输出任务类型。

支持的类别有：

| 类别 | 描述 | 要输出到文件的类别枚举 |
|------|------|---------|
| 需求和方案设计 | issue 文字结构整理工作或需求重写，设计一个业务流程，讨论技术方案且明确表示暂时不需要实现 | issue_refinement |
| 开发 | 要通过修改代码、脚本或仓库里的文档来实现的功能或修复问题。包括明确要求要实现一个 issue，或者完善一个 PR，要求添加新的功能，或者补加新的自动化测试案例，修复失败的 CI，添加架构文档都等 | dev |
| 测试 | 验证产品的某项产品功能，或复现一个可能的缺陷 | test |
| 常规 | 日常杂务，询问仓库统计, GitHub 项目管理，以及日常对话闲聊等各种其他类型的工作 | generic |

可以直接归类的情况：
* 事件来源为 issue 或 PR 的 label 事件，是 dev 类型任务，表示希望创建 PR 来实现需求或优化现有已关联的 PR
* issue 或 PR 的评论内容只有 `/ai-agent` 而没有其他内容，是 dev 类型任务，表示希望你创建 PR 来实现需求或优化现有已关联的 PR
* 在 issue 评论里，问“你有什么看法”、“给我提点建议”、“帮我一起想一想”之类的，是常规对话，也就是 generic 类型

# 2. 执行任务

如果是日常杂务 generic 类工作，你就自己完成它；其他类型的工作你直接跳过，不要执行任何操作，后面会有其他程序负责执行。再次强调，你只执行 generic 类工作。

回答用户的咨询时，结合当前项目仓库中的项目介绍、文档和代码设计等来给出基于事实的回应。

# 3. 任务完成

如果是日常杂务 generic 类工作，是你自己做的，在结束时额外执行以下步骤：
1. 所有步骤完成后，把你的工作结果、结论更新到“触发本次任务的 issue 或 PR”上作为一条新的评论。请使用中文输出。
2. 根据任务结果、结论，按照 `.agents/prompts/base.md` 中的说明添加对“事件来源”的 GitHub 反应：如何任务正常完成，移除 `eyes`，不要添加任何其他反应。
