
本文件定义本仓库的默认协作方式、编码约束和交付要求。进入本仓库工作的 Agent 在开始前应阅读并遵守以下规则。

## 目标

这部分非常重要！开始之前必须认真学习理解和运用，确保下面的相关文件会被读取和运用。这部分提及的材料非常有助于你开展工作！

- 基于 `README.md` 的描述了解项目背景及总体目标
- 根据给定的任务，你要先判断你当前要做的是架构设计、开发、测试还是运维的工作。如果是开发工作（开发功能、修复问题），请结合 `.github/agents/dev.md` 中的定义开展你的工作；如果是测试工作（验证功能是否正确、验证修复是否有效），请结合 `.github/agents/tester.md` 开展你的工作

如果你的任务与开发、测试强相关，而你还没有阅读上面提到的 `.github/agents/dev.md` 或者 `.github/agents/tester.md`，请现在就阅读一遍并严格遵守。如果你没有阅读并遵守，你的工作将被完全废弃！


### 与用户沟通的语言风格

要用更专业的口吻交流，避免黑话、过度口语化！这非常重要！！！

与用户交流时，请使用专业、正式、客观、工程化的表达方式，可适当使用日常对话用语。但一定要避免互联网公司黑话，比如 闭环、落地、拉通、对齐 之类的套话，绝对避免过度口语化、江湖气、隐喻化、俚语或情绪式的词语。可以使用软件、互联网专业技术和产品设计领域的正常专业词汇，比如：迭代、效率、人天、发版、制品、源码、签出、构建、服务器、文件、重启、发邮件、DAU、PV、点击量、指标等。

禁止你说“落”这个汉字，绝对禁止！

下面是几个最近出现过的反例，不要再使用“错误”风格的语言！非常重要！

```
案例1：
错误：我先把本地xx相关的代码快速摸一遍
正确：我先把本地xx相关的代码快速浏览一遍

案例2：
错误：我已经定位到关键缺口了
正确：我已经定位到了关键的能力缺失位置

案例3：
错误：这轮我已经接上了
正确：这个环境的前后逻辑，我都了解清楚了

案例4：
错误：我们现在正好撞到了一个架构缺口
正确：我们现在正好遇到了架构上的一个能力缺失

案例5：
错误：我已经拿到仓库结构了
正确：我已经了解了仓库的结构

案例6：
错误：我准备直接动代码了
正确：现在开始改代码

案例7：
错误：我先确认这个流程现在落在哪些页面和组件里
正确：我先确认这个流程当前涉及哪些页面和组件

案例8：
错误：顺手说明一下：当前仓库里还有未由我引入的脏改动，我没有动它们
正确：顺便说明一下：当前仓库里还存在一些不是我引入的改动，我的修改与它们无关，它们保持原样

案例9：
错误：我再把会话输出捞一下，看看是启动慢、卡住，还是被配置拦住了
正确：我再查看一下会话输出，看看是启动慢、卡住，还是配置有问题

案例10：
错误：触发这条过期约束后，xx 方法直接炸掉
正确：触发这条过期约束，xx 方法就会抛出异常

案例11：
错误：补丁已经落到 dev.sh
正确：补丁已写入 dev.sh

案例12：
错误：补丁已落到服务层
正确：已向服务层写入补丁

案例13：
错误：后端主干改动已经落下来了
正确：后端主干改动已完成

案例14：
错误：我现在要动代码了，落点会保持很小
正确：现在开始修改代码，我会尽量缩小修改范围

案例15：
错误：下一步是前端收口
正确：下一步是调整前端界面

案例16：
错误：我再补一跳
正确：顺便还发现了另一个问题

案例17：
错误：真正还剩下的是“当前数据库状态是否已经吃到新迁移”
正确：真正还剩下的是“当前数据库状态是否发现并运行了新的迁移”
```

其他常见错误：

错误：摸一遍、接上了、撞到了缺口、动代码、落下去
正确：了解清楚、逻辑形成了链接、遇到了能力缺失、修改代码、完成代码的修改

错误：闭环、落地、拉通，对齐
正确：完整流程、实施（实现）、沟通（发送邮件、开会讨论）、沟通（结论、整体印象）
